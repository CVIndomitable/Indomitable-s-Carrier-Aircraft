# 技术挑战分析

## 设计优势

1. **架构清晰**
   - 子系统划分合理（核心执行层/决策控制层/特殊功能层/界面层）
   - 依赖关系明确
   - 状态机模式适合复杂 AI 行为

2. **数据驱动设计**
   - 机型能力完全由配置表决定
   - 新增机型无需改代码
   - 扩展性极好

3. **功能完整**
   - 涵盖了舰载机的核心玩法（轰炸、鱼雷、空战、侦察）
   - 编组系统层级合理
   - 火控系统符合现代 RTS 逻辑

## 技术挑战详解

### 1. AI 与路径规划（⭐⭐⭐⭐⭐ 高难度）

#### 三坐标计算系统
```
目标坐标 -> 弹着点 -> 投弹坐标
```

**挑战**：
- 需要精确的物理计算（弹道时间、飞机速度、提前量）
- 不同弹药类型需要不同的计算公式
- 动态修正机制（目标移动时重新计算）

#### 飞行物理模拟
- 俯冲、拉起、盘旋的平滑运动
- 航向调整算法（不能直角转弯）
- 高度控制（俯冲轰炸需要精确高度管理）

#### 空战 AI
- 追击路径规划（保持额定距离）
- 放弃条件判断（追不上、超出范围）
- 前射型与自卫炮塔的不同行为

### 2. 客户端/服务端同步（⭐⭐⭐⭐ 中高难度）

**需要同步的数据**：
- 飞机状态（10+ 种状态）
- 锁定目标列表（火控系统）
- 编组信息（小队/中队/大队归属）
- 弹药数量（双武器槽）
- 玩家配置（投弹数量、档位等）

**实现建议**：
- 使用 NeoForge 的 `EntityDataAccessor` 系统
- 关键状态变化时发送自定义网络包
- 控制面板 GUI 的网络通信架构需要提前设计

### 3. 性能优化（⭐⭐⭐⭐ 高难度）

**潜在性能瓶颈**：
- 多架飞机同时运行（每架都有状态机 + 弹道计算）
- 区块强制加载（多架长机可能加载大量区块）
- 目标搜索（每 tick 扫描范围内所有实体）
- 弹药实体（制导炸弹需要持续追踪）

**优化策略**：
- 限制同时运行的飞机数量
- 目标搜索使用缓存（不需要每 tick 重新扫描）
- 弹道计算可以降频（每 5-10 tick 更新一次）
- 区块加载使用 Ticket 系统，及时释放

### 4. GUI 复杂度（⭐⭐⭐⭐ 中高难度）

**控制面板需要实现**：
- 实时小地图（以长机为中心）
- 飞机位置标注（不同机型不同图标）
- 航线显示
- 多层级配置界面
- 弹药装载管理

**技术要求**：
- 自定义渲染逻辑（地图绘制）
- 复杂的 GUI 布局（多标签页？）
- 大量的网络通信（实时更新地图数据）

### 5. 区块加载管理（⭐⭐⭐ 中难度）

**关键点**：
- 长机作为区块加载锚点
- 需要追踪所有长机的位置
- 长机被击落时转移加载点
- 注意不要造成区块加载泄漏

**实现方式**：
- 使用 `ServerLevel.setChunkForced()` 或 Ticket 系统

| 长机与区块加载 | ⭐⭐⭐ | 1-2 周 |
| 控制面板 GUI | ⭐⭐⭐⭐ | 3-4 周 |
| 客户端同步调试 | ⭐⭐⭐⭐ | 2-3 周 |
| 性能优化 | ⭐⭐⭐⭐ | 2-3 周 |
